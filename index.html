📖 Platform Guide

Solutions4Society — Attendance Management Platform

🏢 Three Roles
🏢
Company (Solutions4Society)
Creates and manages all societies. Can view all attendance across all societies. Exports full reports. Enables/disables societies.
🏘️
Society Admin
Logs in with credentials sent by Solutions4Society. Manages their own staff — add, edit, enable/disable. Views their own attendance only. Cannot see other societies.
👤
Staff
Logs in with credentials sent by society admin. Can only check in/out and view their own history. Cannot see other staff or society settings.
🚀 First-Time Company Setup
⚙️
Step 1 — Get your Company UID
Log in once with your Solutions4Society email. Open browser console (F12). Copy the UID shown as "Company UID: xxx". Paste it into COMPANY_UID in the HTML file. Re-upload.
🛡️
Step 2 — Update Firestore Rules
Copy the rules below into Firebase Console → Firestore → Rules → Publish.
rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { match /userRoles/{uid} { allow read: if request.auth != null && request.auth.uid == uid; allow write: if request.auth != null && request.auth.uid == resource.data.get('companyUid', uid); } match /societies/{societyId} { allow read: if request.auth != null && (get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'company' || get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.societyId == societyId); allow write: if request.auth != null && get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'company'; } match /societies/{societyId}/staff/{staffId} { allow read: if request.auth != null && (get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'company' || get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.societyId == societyId); allow write: if request.auth != null && (get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'company' || get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'society'); } match /societies/{societyId}/attendance/{logId} { allow read: if request.auth != null && (get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.role == 'company' || get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.societyId == societyId); allow write: if request.auth != null && get(/databases/$(database)/documents/userRoles/$(request.auth.uid)).data.societyId == societyId; } } }
📧 Email Flows
🏘️
Society Created
Society admin receives their portal URL, login email, and temporary password. They must change the password on first login.
👤
Staff Added
Staff member receives the portal link, login email, and temporary password. They must change it on first login.
✅
Check-In / Check-Out
Society admin's email receives an automatic notification every time a staff member checks in or out. No action needed.
🔐 Security
🔒
Data Isolation
Society data lives in /societies/{societyId}/staff and /societies/{societyId}/attendance subcollections. Firestore rules enforce that a society can only read its own subcollection path — even if someone manually modifies the URL, they get a permission error, not another society's data.
👤
Role-Based Access
Every user's role (company / society / staff) is stored in /userRoles/{uid} and checked server-side by Firestore rules. The client cannot fake a role — even by editing JS in the browser, data access is blocked at the database layer.
